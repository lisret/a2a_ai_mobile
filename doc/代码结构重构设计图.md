# 代码结构重构设计图

## 📋 设计目标

1. **明确模块职责** - 每个模块的作用清晰明了
2. **模块化设计** - 高内聚、低耦合
3. **易于扩展** - 新增功能无需大改
4. **常量集中管理** - 所有硬编码配置统一管理

│   └── DialogueService.ts          # 对话API服务

**职责**:
- 任务执行流程编排

### 5. Config Layer (配置管理层) ⭐ 新增

**职责**: 集中管理所有硬编码常量

```
shared/constants/
├── index.ts                      # 统一导出入口
├── task.config.ts                # 任务执行配置
│   ├── TASK_CONFIG               # 任务基础配置
│   ├── TASK_TIMEOUT_CONFIG       # 超时配置
│   └── TASK_RETRY_CONFIG         # 重试配置
├── api.config.ts                 # API配置
│   ├── API_PROVIDERS             # API提供商列表
│   ├── API_REQUEST_CONFIG        # 请求配置
│   └── API_TIMEOUT_CONFIG        # API超时配置
├── ui.config.ts                  # UI配置
│   ├── COLORS                    # 颜色主题
│   ├── SHADOWS                   # 阴影样式
│   ├── UI_DIMENSIONS             # 尺寸常量
│   └── UI_ANIMATIONS             # 动画配置
├── storage.config.ts             # 存储配置
│   └── STORAGE_KEYS              # 存储键名
├── app.config.ts                 # 应用配置
│   ├── ADB_CONFIG                # ADB配置
│   ├── ACCESSIBILITY_CONFIG      # 无障碍配置
│   └── AUTOGLM_CONFIG            # AutoGLM配置
└── messages.config.ts            # 消息配置
    ├── ERROR_MESSAGES            # 错误消息
    └── UI_MESSAGES               # UI消息
```

**配置分类说明**:

1. **task.config.ts** - 任务执行相关
   - 最大步数、超时时间、重试次数、等待时间等

2. **api.config.ts** - API调用相关
   - API提供商、请求参数、超时配置等

3. **ui.config.ts** - UI展示相关
   - 颜色、尺寸、样式、动画等

4. **storage.config.ts** - 存储相关
   - AsyncStorage键名、缓存键名等

5. **app.config.ts** - 应用功能相关
   - ADB配置、无障碍配置、应用映射等

6. **messages.config.ts** - 消息文本相关
   - 错误提示、UI文本、确认消息等

## 🎯 扩展点设计

### 1. 新增模型支持
```
core/engine/dialogue/
└── [新模型名称]/
    ├── prompts/
    │   └── PromptProvider.ts    # 实现 IPromptProvider
    └── parsers/
        └── Parser.ts            # 实现 IParser
```
**步骤**:
1. 实现 `IPromptProvider` 接口
2. 实现 `IParser` 接口
3. 在 `factory.ts` 中注册新模型

### 2. 新增能力支持
```
core/ability/
└── [新能力名称]/
    ├── [能力名称]Service.ts
    ├── [能力名称]Capability.ts
    └── index.ts
```
**步骤**:
1. 实现 `ICapability` 接口
2. 在 `index.ts` 中注册新能力
3. 能力管理器自动识别

### 3. 新增功能模块
```
features/
└── [新功能名称]/
    ├── screens/
    ├── components/
    ├── services/
    └── hooks/
```
**步骤**:
1. 创建功能目录
2. 按标准结构组织文件
3. 在导航中注册新页面

### 4. 新增配置项
```
shared/constants/
└── [分类].config.ts
    └── [配置名称]: { ... }
```
**步骤**:
1. 在对应的配置文件中添加
2. 从 `index.ts` 导出
3. 在代码中引用

## 📐 常量配置使用示例

### 配置定义 (shared/constants/task.config.ts)
```typescript
export const TASK_CONFIG = {
  MAX_STEPS: 50,
  MAX_CONSECUTIVE_ERRORS: 3,
  API_TIMEOUT_MS: 60000,
  SCREENSHOT_TIMEOUT_MS: 5000,
  ACTION_WAIT_DURATION: 300,
} as const;
```

### 配置使用
```typescript
// ✅ 正确：从配置中导入
import { TASK_CONFIG } from '@shared/constants';

const maxSteps = TASK_CONFIG.MAX_STEPS;

// ❌ 错误：硬编码
const maxSteps = 50;
```

## 🎨 设计原则

1. **单一职责原则** - 每个模块只负责一个功能域
2. **依赖倒置原则** - 通过接口定义依赖关系
3. **开闭原则** - 对扩展开放，对修改关闭
4. **接口隔离原则** - 使用接口而非具体实现
5. **配置集中原则** - 所有硬编码配置统一管理
6. **模块化原则** - 按功能域组织代码，高内聚低耦合
