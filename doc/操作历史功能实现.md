# 操作历史功能实现

## 功能概述

将最近的操作历史包含在提示词中，帮助模型更好地理解当前状态和下一步操作。这是参考 Open-AutoGLM 框架的重要优化。

## 实现内容

### 1. 操作历史格式化

在 `AutoGLMService.ts` 中添加了 `formatActionHistory()` 方法：

- **功能**：将操作步骤列表格式化为对模型友好的文本
- **参数**：
  - `steps`: 操作步骤列表
  - `maxSteps`: 最多包含的步骤数（默认10步）
- **返回**：格式化的操作历史文本

**格式化示例**：
```
操作历史（最近5步）：
步骤 1: 点击 (540, 960) (坐标: 540, 960)
步骤 2: 输入: 测试文本 (文本: 测试文本)
步骤 3: 滑动 (100,200 到 300,400) (从 100,200 到 300,400)
步骤 4: 启动应用: com.example.app (应用: com.example.app)
步骤 5: 返回
```

### 2. 集成到模型API调用

在 `getAction()` 方法中：

- **新增参数**：`actionHistory` - 操作历史列表
- **处理逻辑**：
  1. 如果有操作历史，调用 `formatActionHistory()` 格式化
  2. 将格式化的操作历史追加到用户消息文本中
  3. 模型可以看到最近的操作步骤，更好地理解当前状态

**示例用户消息**：
```
继续执行任务

操作历史（最近5步）：
步骤 1: 点击 (540, 960) (坐标: 540, 960)
步骤 2: 输入: 搜索关键词 (文本: 搜索关键词)
步骤 3: 滑动 (从 500,1500 到 500,500)
步骤 4: 点击 (800, 1200) (坐标: 800, 1200)
步骤 5: 等待 2000ms
```

### 3. 传递操作历史

在以下位置传递操作历史：

1. **TaskExecutionHeadless.ts**（后台任务执行）
   ```typescript
   autoGLMService.getAction(
     screenshotUri,
     instruction,
     model,
     onStreamUpdate,
     conversationHistory,
     steps // 传递操作历史
   );
   ```

2. **useTaskExecution.ts**（前台任务执行）
   ```typescript
   autoGLMService.getAction(
     screenshotUri!,
     taskInstruction,
     model,
     onStreamUpdateCallback,
     conversationHistoryRef.current,
     steps // 传递操作历史
   );
   ```

## 功能特点

### 1. 智能截取
- 只包含最近的步骤（默认10步），避免提示词过长
- 可以通过 `maxSteps` 参数调整

### 2. 详细信息
- 包含操作类型和描述
- 对于点击操作，包含坐标信息
- 对于滑动操作，包含起始和结束坐标
- 对于输入操作，包含文本内容（截断长文本）
- 对于启动应用，包含应用包名

### 3. 自动集成
- 无需手动管理，自动从步骤列表中提取
- 与对话历史配合使用，提供完整的上下文

## 优势

1. **更好的上下文理解**
   - 模型可以看到之前执行的操作
   - 避免重复操作或错误操作
   - 更好地规划下一步

2. **提高任务成功率**
   - 模型可以理解当前状态
   - 避免在错误的页面执行操作
   - 更好地处理复杂任务流程

3. **符合参考框架**
   - 与 Open-AutoGLM 框架的设计理念一致
   - 充分利用操作历史信息

## 使用示例

### 场景1：搜索任务

**操作历史**：
```
步骤 1: 启动应用: com.example.browser
步骤 2: 点击 (540, 960) (坐标: 540, 960)  # 点击搜索框
步骤 3: 输入: 咖啡店 (文本: 咖啡店)
步骤 4: 点击 (900, 960) (坐标: 900, 960)  # 点击搜索按钮
步骤 5: 滑动 (从 540,1500 到 540,500)  # 滚动查看结果
```

**模型理解**：
- 已经打开了浏览器
- 已经输入了搜索关键词
- 已经点击了搜索按钮
- 当前正在查看搜索结果
- 下一步应该是选择具体的咖啡店

### 场景2：购物任务

**操作历史**：
```
步骤 1: 启动应用: com.example.shop
步骤 2: 点击 (540, 400) (坐标: 540, 400)  # 点击搜索框
步骤 3: 输入: 手机 (文本: 手机)
步骤 4: 点击 (900, 960) (坐标: 900, 960)  # 点击搜索
步骤 5: 滑动 (从 540,1500 到 540,500)  # 浏览商品
步骤 6: 点击 (540, 800) (坐标: 540, 800)  # 点击商品
步骤 7: 滑动 (从 540,1500 到 540,500)  # 查看商品详情
步骤 8: 点击 (540, 1800) (坐标: 540, 1800)  # 点击加入购物车
```

**模型理解**：
- 已经搜索了商品
- 已经选择了商品
- 已经查看了商品详情
- 已经加入了购物车
- 下一步应该是去购物车结算

## 配置建议

### 调整历史步数

如果需要包含更多或更少的步骤，可以修改 `formatActionHistory()` 的 `maxSteps` 参数：

```typescript
// 包含最近20步
const historyText = this.formatActionHistory(actionHistory, 20);
```

### 调整历史格式

如果需要自定义历史格式，可以修改 `formatActionHistory()` 方法的实现。

## 注意事项

1. **性能考虑**
   - 操作历史会增加提示词长度
   - 默认只包含最近10步，平衡上下文和性能

2. **Token限制**
   - 如果模型有token限制，可能需要减少历史步数
   - 或者只包含关键操作步骤

3. **隐私考虑**
   - 操作历史可能包含敏感信息（如输入文本）
   - 对于敏感操作，可以考虑不包含详细信息

## 后续优化建议

1. **智能过滤**
   - 只包含关键操作步骤
   - 过滤掉等待、返回等辅助操作

2. **操作摘要**
   - 对于长任务，提供操作摘要
   - 而不是列出所有步骤

3. **状态标记**
   - 标记操作是否成功
   - 帮助模型理解哪些操作生效了

4. **页面识别**
   - 结合页面截图，识别当前页面
   - 与操作历史结合，提供更准确的上下文
