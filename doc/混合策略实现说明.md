# 混合多重策略实现说明

## 概述

已根据**方案4：混合多重策略**实现了获取所有应用包名的功能，**完全不使用root权限**，使用标准的Android API。

## 实现内容

### 1. Kotlin 原生模块实现

在 `AccessibilityActionModule.kt` 中新增了 `getAllInstalledAppsHybrid()` 方法，实现了以下策略：

#### 策略1: getInstalledPackages（最标准）
- 使用 `PackageManager.getInstalledPackages()` 获取所有已安装包
- 包含系统应用和用户应用
- 获取应用名称和启动Intent信息

#### 策略2: getInstalledApplications（补充）
- 当策略1获取的应用数量少于50个时自动触发
- 使用 `PackageManager.getInstalledApplications()` 补充可能遗漏的应用
- 确保获取到所有应用

#### 策略3: Intent查询（补充可启动应用）
- 查询 `ACTION_MAIN + CATEGORY_LAUNCHER` Intent
- 确保获取到所有有启动入口的应用
- 补充前两个策略可能遗漏的可启动应用

#### 策略4: MATCH_UNINSTALLED_PACKAGES（兜底）
- 当前面策略都失败或获取的应用少于10个时触发
- 使用 `MATCH_UNINSTALLED_PACKAGES` 标志
- 可以获取到已卸载但保留数据的应用

### 2. TypeScript 服务层更新

在 `ADBAppMappingService.ts` 中更新了 `fetchAllInstalledApps()` 方法：

- **优先使用混合策略方法** (`getAllInstalledAppsHybrid`)
- 如果混合策略失败，自动回退到标准方法 (`getAllInstalledApps`)
- 保持原有的缓存机制不变

## 关键特性

### ✅ 不使用root权限
所有方法都使用标准的Android API，不需要任何特殊权限：
- `PackageManager.getInstalledPackages()` - 标准API
- `PackageManager.getInstalledApplications()` - 标准API
- `PackageManager.queryIntentActivities()` - 标准API
- `PackageManager.getLaunchIntentForPackage()` - 标准API

### ✅ 自动去重
使用 `Set<String>` 记录已添加的包名，确保不会重复添加

### ✅ 异常处理
- 每个策略独立 try-catch，一个失败不影响其他策略
- 每个应用单独处理，单个应用失败不影响整体

### ✅ 智能补充
- 根据获取的应用数量自动决定是否执行补充策略
- 避免不必要的性能开销

### ✅ 日志记录
记录使用的策略和方法，便于调试和优化

## 使用方式

### 自动使用（推荐）

服务层会自动优先使用混合策略方法，无需修改调用代码：

```typescript
import { appMappingService } from './abilityModule/code/ADB/ADBAppMappingService';

// 自动使用混合策略
const mapping = await appMappingService.getAppMapping();
```

### 手动调用原生方法

如果需要直接调用原生方法：

```typescript
import { NativeModules } from 'react-native';

const { AccessibilityActionModule } = NativeModules;

// 使用混合策略方法
const mapping = await AccessibilityActionModule.getAllInstalledAppsHybrid();

// 或使用标准方法
const mapping = await AccessibilityActionModule.getAllInstalledApps();
```

## 性能考虑

1. **缓存机制**: 应用列表会缓存24小时，避免频繁查询
2. **条件执行**: 如果策略1成功获取足够应用，会自动跳过其他策略
3. **去重处理**: 使用Set进行高效去重

## 兼容性

- ✅ 支持所有 Android 版本（API 1+）
- ✅ 兼容所有厂商定制系统（MIUI、ColorOS、EMUI等）
- ✅ 不需要root权限
- ✅ 不需要特殊权限（只需要基本的PackageManager访问权限）

## 测试建议

1. **不同Android版本**: 在Android 5.0、8.0、11.0、13.0等版本上测试
2. **不同厂商**: 在小米、华为、OPPO、vivo等设备上测试
3. **应用数量**: 在应用数量较少（<50）和较多（>200）的设备上测试
4. **异常情况**: 测试某些应用无法获取名称的情况

## 注意事项

1. **性能**: 如果设备上应用很多（>300），首次获取可能需要几秒钟
2. **缓存**: 应用列表变化不频繁，24小时缓存是合理的
3. **系统应用**: 默认包含系统应用，如果需要过滤可以在策略中添加判断
4. **同名应用**: 如果多个应用有相同名称，后面的会覆盖前面的（使用包名作为key可以避免）

## 后续优化建议

1. **并行执行**: 可以考虑并行执行多个策略，提高速度
2. **厂商适配**: 针对特定厂商可以优化策略执行顺序
3. **增量更新**: 可以只查询新增或删除的应用，而不是全量查询
4. **过滤选项**: 可以添加选项来过滤系统应用或特定类型的应用
