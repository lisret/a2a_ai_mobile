# 任务暂停/恢复机制调研

## 调研目标

调研当应用从后台打开（回到前台）时，是否能够暂停正在执行的任务，以及如何实现暂停/恢复机制。

## 当前实现分析

### 1. 任务执行架构

当前任务执行采用以下架构：

```
前台 UI (React Native)
    ↓
后台任务启动 (useTaskExecutionWithBackground)
    ↓
Headless JS 任务 (TaskExecutionHeadless.ts)
    ↓
执行循环 (while loop)
    ├── 截图
    ├── 调用模型 API
    ├── 解析操作指令
    └── 执行操作（点击、滑动等）
```

## 暂停/恢复机制可行性分析

### ✅ 技术可行性

#### 1. React Native Headless JS 限制

**限制：**
- Headless JS 本身不提供暂停/恢复 API
- Headless JS 任务在独立的 JavaScript 线程中运行
- 无法直接暂停 JavaScript 执行

**解决方案：**
- 使用状态标志（类似 `shouldCancel`）控制任务执行
- 在任务循环中检查暂停状态
- 暂停时停止执行新步骤，但保持任务状态

#### 2. 实现方案

**方案 A：基于状态标志的暂停（推荐）**

```typescript
// 暂停标志
let isPaused = false;
let pauseResumeListener: any = null;

// 监听暂停/恢复事件
pauseResumeListener = DeviceEventEmitter.addListener('TaskPauseRequested', (data) => {
  if (data.taskId === taskId || data.taskId === 'current') {
    isPaused = true;
    console.log('[Headless JS] 任务已暂停');
    // 发送暂停事件到前台
    DeviceEventEmitter.emit('TaskPaused', { taskId });
  }
});

DeviceEventEmitter.addListener('TaskResumeRequested', (data) => {
  if (data.taskId === taskId || data.taskId === 'current') {
    isPaused = false;
    console.log('[Headless JS] 任务已恢复');
    DeviceEventEmitter.emit('TaskResumed', { taskId });
  }
});

// 在执行循环中检查暂停状态
while (step < maxSteps) {
  // 检查取消
  if (shouldCancel) {
    break;
  }
  
  // 检查暂停 - 等待恢复
  while (isPaused && !shouldCancel) {
    await new Promise(resolve => setTimeout(resolve, 100));
  }
  
  // 执行任务步骤...
}
```

**优点：**
- ✅ 实现简单，基于现有取消机制扩展
- ✅ 不需要修改底层架构
- ✅ 可以保存任务状态，恢复后继续执行

**缺点：**
- ⚠️ 暂停期间仍然占用资源（等待循环）
- ⚠️ 如果应用被系统杀死，暂停状态会丢失

**方案 B：保存状态并停止执行**

```typescript
// 暂停时保存当前状态
if (isPaused) {
  // 保存任务状态到存储
  await taskHistoryService.saveTask({
    id: taskId,
    status: 'paused',
    currentStep: step,
    // ... 其他状态
  });
  
  // 停止执行，等待恢复事件
  return; // 或抛出特殊异常
}

// 恢复时从存储加载状态
const savedTask = await taskHistoryService.getTaskById(taskId);
if (savedTask?.status === 'paused') {
  step = savedTask.currentStep;
  // 恢复执行
}
```

**优点：**
- ✅ 暂停时真正停止执行，节省资源
- ✅ 状态持久化，应用重启后可以恢复

**缺点：**
- ⚠️ 实现复杂度较高
- ⚠️ 需要处理状态序列化/反序列化
- ⚠️ Headless JS 停止后需要重新启动

### ❌ 技术限制

#### 1. Headless JS 生命周期限制

- Headless JS 任务一旦启动，会持续运行直到完成或取消
- 无法直接"暂停" Headless JS 线程
- 只能通过标志位控制执行流程

#### 2. Android 后台限制

- Android 系统可能会限制后台任务执行
- 某些厂商 ROM 可能会杀死后台任务
- 暂停的任务可能被系统回收

#### 3. 状态同步问题

- 暂停/恢复状态需要在 Headless JS 和前台 UI 之间同步
- 如果应用被杀死，状态可能丢失
- 需要持久化存储来保存暂停状态

## 推荐实现方案

### 方案：基于状态标志的暂停（方案 A）

**实现步骤：**

1. **添加暂停/恢复事件**
   ```typescript
   // 在 TaskExecutionHeadless.ts 中添加
   DeviceEventEmitter.emit('TaskPauseRequested', { taskId });
   DeviceEventEmitter.emit('TaskResumeRequested', { taskId });
   ```

2. **在任务循环中检查暂停状态**
   ```typescript
   while (step < maxSteps) {
     // 检查取消
     if (shouldCancel) break;
     
     // 检查暂停 - 等待恢复
     while (isPaused && !shouldCancel) {
       await new Promise(resolve => setTimeout(resolve, 100));
     }
     
     // 执行任务步骤...
   }
   ```

3. **在 AppNavigator 或 HomeScreen 中监听 AppState**
   ```typescript
   useEffect(() => {
     const subscription = AppState.addEventListener('change', (nextAppState) => {
       if (nextAppState === 'active' && executing) {
         // 应用回到前台，暂停任务
         DeviceEventEmitter.emit('TaskPauseRequested', { taskId: 'current' });
       } else if (nextAppState === 'background' && executing && isPaused) {
         // 应用切换到后台，恢复任务
         DeviceEventEmitter.emit('TaskResumeRequested', { taskId: 'current' });
       }
     });
     return () => subscription.remove();
   }, [executing, isPaused]);
   ```

4. **UI 显示暂停状态**
   - 在 ExecutionCard 中显示"已暂停"状态
   - 提供"恢复"按钮
   - 显示暂停原因（如"应用回到前台"）

### 注意事项

1. **资源消耗**
   - 暂停期间仍然运行等待循环，会消耗少量 CPU
   - 建议使用较长的等待间隔（100-500ms）

2. **状态持久化**
   - 如果需要在应用重启后恢复，需要保存暂停状态到 AsyncStorage
   - 恢复时需要重新启动 Headless JS 任务

3. **用户体验**
   - 暂停时应该给用户明确的提示
   - 提供手动恢复的选项
   - 考虑自动恢复的场景（如应用回到后台）

4. **边界情况**
   - 任务在暂停期间被系统杀死
   - 暂停时收到取消请求
   - 多个任务同时执行时的暂停逻辑

## 结论

### ✅ 可行性

**技术上可行：** 可以通过状态标志实现暂停/恢复机制，类似于现有的取消机制。

### ⚠️ 限制

1. **不是真正的暂停：** 只是停止执行新步骤，等待循环仍在运行
2. **资源消耗：** 暂停期间仍会消耗少量资源
3. **状态持久化：** 需要额外实现才能支持应用重启后恢复

### 📋 建议

1. **短期方案：** 实现基于状态标志的暂停（方案 A）
   - 实现简单，快速上线
   - 满足基本需求

2. **长期方案：** 考虑实现状态持久化（方案 B）
   - 支持应用重启后恢复
   - 更好的资源管理

3. **用户体验优化：**
   - 提供手动暂停/恢复按钮
   - 显示暂停原因和状态
   - 考虑自动暂停的场景（如应用回到前台）
