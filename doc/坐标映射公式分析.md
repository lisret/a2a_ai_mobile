# 坐标映射公式分析

## 当前坐标映射公式

```typescript
// 从 0-999 坐标系映射到实际屏幕分辨率
mappedX = Math.round((x / 1000) * screenSize.width)
mappedY = Math.round((y / 1000) * screenSize.height)
```

## 公式分析

### 公式假设

1. **模型坐标系**：0-999（共1000个值：0, 1, 2, ..., 999）
2. **映射关系**：线性映射，模型坐标的百分比 = 实际坐标的百分比
3. **截图分辨率 = 屏幕分辨率**：假设截图和屏幕分辨率一致

### 公式正确性验证

#### 场景1：标准竖屏设备（1080x1920）

- 模型返回：`do(action="Tap", element=[500,500])`
- 映射计算：
  - `mappedX = Math.round((500 / 1000) * 1080) = 540`
  - `mappedY = Math.round((500 / 1000) * 1920) = 960`
- **结果**：✓ 正确（中心点）

#### 场景2：高分辨率设备（1440x3200）

- 模型返回：`do(action="Tap", element=[500,500])`
- 映射计算：
  - `mappedX = Math.round((500 / 1000) * 1440) = 720`
  - `mappedY = Math.round((500 / 1000) * 3200) = 1600`
- **结果**：✓ 正确（中心点）

#### 场景3：横屏设备（1920x1080）

- 模型返回：`do(action="Tap", element=[500,500])`
- 映射计算：
  - `mappedX = Math.round((500 / 1000) * 1920) = 960`
  - `mappedY = Math.round((500 / 1000) * 1080) = 540`
- **结果**：✓ 正确（中心点）

## 潜在问题分析

### 问题1：截图分辨率 vs 屏幕分辨率不一致

**问题描述**：
- 如果截图被压缩或缩放，截图分辨率 ≠ 屏幕分辨率
- 模型看到的是截图，但坐标映射使用的是屏幕分辨率
- 这会导致坐标映射错误

**当前实现检查**：
- ✅ 截图保持原始分辨率（只进行JPEG压缩，不缩放）
- ✅ 使用 `getScreenSize()` 获取实际屏幕分辨率
- ✅ 截图分辨率应该等于屏幕分辨率

**风险**：
- ⚠️ 如果未来对截图进行缩放优化，需要同时更新坐标映射逻辑

### 问题2：屏幕方向变化

**问题描述**：
- 如果设备横屏，但截图是竖屏（或反之）
- 坐标映射会出错

**当前实现检查**：
- ✅ `getScreenSize()` 使用 `currentWindowMetrics` 或 `getRealMetrics`，应该返回当前方向的分辨率
- ⚠️ 需要确认截图方向是否与屏幕方向一致

### 问题3：宽高比差异

**问题描述**：
- 不同设备的宽高比不同（16:9, 18:9, 20:9等）
- 模型在0-999坐标系中，相对位置是准确的
- 但绝对坐标映射可能因为宽高比不同而有细微差异

**影响**：
- 对于相对位置（如"屏幕中心"），公式是正确的
- 对于绝对位置，由于宽高比不同，可能会有1-2像素的误差（可接受）

## 公式验证

   - 模型：`[0, 0]`
   - 映射：`(0, 0)` ✓

## 结论

### ✅ 公式本身是正确的

当前公式 `mappedX = Math.round((x / 1000) * screenSize.width)` 是正确的，因为：
1. 模型使用相对坐标系（0-999），表示相对位置百分比
2. 映射到实际分辨率时，保持相同的相对位置
3. 这是Open-AutoGLM框架的标准做法

### ⚠️ 需要注意的问题

1. **截图分辨率必须等于屏幕分辨率**
   - 如果截图被缩放，需要同时缩放坐标
   - 当前实现：截图保持原始分辨率 ✓

2. **屏幕方向必须一致**
   - 截图方向和屏幕方向必须一致
   - 需要验证 `getScreenSize()` 返回的是当前方向的分辨率

3. **分辨率缓存问题**
   - 如果屏幕方向改变，缓存的分辨率可能过期
   - 需要清除缓存或实时获取
