## 应用启动方式对比

### Open-AutoGLM-Android (GitHub项目)

#### 启动方式
根据项目描述和常见实现模式，该项目主要使用：

1. **无障碍服务直接启动**
   - 通过无障碍服务（AccessibilityService）获取应用启动 Intent
   - 使用 `PackageManager.getLaunchIntentForPackage()` 获取启动 Intent
   - 在无障碍服务中直接调用 `startActivity()` 启动应用

2. **可能的实现方式**（基于原生 Android 最佳实践）：
```kotlin
// 在无障碍服务中启动应用
val pm = packageManager
val intent = pm.getLaunchIntentForPackage(packageName)
intent?.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
startActivity(intent)
```

#### 特点
- ✅ **单一方案**：主要依赖无障碍服务启动
- ✅ **简单直接**：不需要复杂的降级策略
- ⚠️ **依赖性强**：完全依赖无障碍服务，如果服务被禁用则无法启动

---

### 当前项目 (React Native项目)

#### 启动方式（多级降级策略）

**步骤1：通用方案（无障碍服务）**
```kotlin
// 使用 getLaunchIntentForPackage（最常用）
val intent = pm.getLaunchIntentForPackage(packageName)
intent?.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
intent?.addFlags(Intent.FLAG_ACTIVITY_RESET_TASK_IF_NEEDED)
intent?.addFlags(Intent.FLAG_ACTIVITY_CLEAR_TOP)
reactApplicationContext.startActivity(intent)
```

**步骤2：品牌特定方案（无障碍服务 + 品牌优化）**
- 检测设备品牌（小米、vivo、OPPO、华为、三星等）
- 根据不同品牌添加特定的 Intent 标志和参数
- 例如：
  - **MIUI**：添加 `miui_start_activity` 参数和特殊标志 `0x10000000`
  - **vivo**：添加 `vivo_start_activity` 参数和特殊标志 `0x20000000`
  - **ColorOS/OxygenOS**：添加 `FLAG_ACTIVITY_RESET_TASK_IF_NEEDED` 标志

**步骤3：ADB命令降级（非root）**
```bash
# 方法1: 使用 am start 命令（不指定Activity）
am start -a android.intent.action.MAIN -c android.intent.category.LAUNCHER {packageName}

# 方法2: 使用 am start 命令（指定包名）
am start -a android.intent.action.MAIN -c android.intent.category.LAUNCHER -n {packageName}/

# 方法3: 使用 am start 命令（尝试MainActivity）
am start -a android.intent.action.MAIN -c android.intent.category.LAUNCHER -n {packageName}/.MainActivity
```

#### 特点
- ✅ **多级降级**：通用方案 → 品牌特定方案 → ADB命令
- ✅ **高兼容性**：针对不同品牌设备进行优化
- ✅ **容错性强**：即使无障碍服务失败，还有ADB命令作为备选
- ✅ **无需root**：所有方案都不需要root权限

## 权限要求对比

### Open-AutoGLM-Android (GitHub项目)

#### 必需权限
根据项目 README 和常见实现：

1. **无障碍服务权限**（必需）
   - `android.permission.BIND_ACCESSIBILITY_SERVICE`
   - 用户需要手动在系统设置中开启

2. **网络访问权限**（必需）
   - `android.permission.INTERNET`
   - 用于访问 AutoGLM API

3. **查询所有应用权限**（用于启动应用）
   - `android.permission.QUERY_ALL_PACKAGES`（Android 11+）
   - 用于查询和启动其他应用

4. **可能的额外权限**（根据实现）：
   - 可能需要 **Shizuku 服务**获取更高权限（用于某些系统级操作）
   - 截图权限（Android 11+）

#### Shizuku 服务说明

**Shizuku 是什么？**

Shizuku 是一个开源项目，旨在为非 Root 用户提供访问 Android 系统 API 的途径。它通过在设备上启动一个具有高权限的服务，使普通应用程序能够直接调用系统级 API，而无需获取 Root 权限。

**Shizuku 的主要特点：**
- ✅ **非 Root 环境下的系统 API 访问**：允许应用程序在非 Root 设备上调用通常需要高权限的系统 API
- ✅ **多种启动方式**：支持通过无线调试（ADB）或 Root 权限启动服务
- ✅ **广泛的兼容性**：兼容多种 Android 设备和系统版本

**使用 Shizuku 的步骤：**
1. 启用开发者选项
2. 启用无线调试（或使用 Root 权限）
3. 安装并配置 Shizuku 应用
4. 通过无线调试或 Root 方式启动 Shizuku 服务
5. 在应用中使用 Shizuku API 调用系统级功能

**为什么可能需要 Shizuku？**
- 某些系统级操作需要高权限（如修改系统设置、强制停止应用等）
- 在 Android 11+ 上，某些 API 需要特殊权限才能访问
- 如果无障碍服务无法满足所有需求，可能需要 Shizuku 作为补充

**注意：**
- Shizuku 需要用户手动安装和配置，增加了使用复杂度
- 不同品牌设备可能需要不同的设置步骤
- 不是所有设备都支持无线调试方式启动

#### 权限特点
- ⚠️ **可能需要高权限**：如果使用 Shizuku，需要用户手动安装和授权
- ⚠️ **依赖性强**：主要依赖无障碍服务权限
- ⚠️ **使用复杂**：需要用户手动配置 Shizuku，增加了使用门槛

### 当前项目 (React Native项目)

#### 必需权限

**AndroidManifest.xml 中声明的权限：**

1. **基本权限**
   ```xml
   <uses-permission android:name="android.permission.INTERNET" />
   <uses-permission android:name="android.permission.WRITE_EXTERNAL_STORAGE" />
   <uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE" />
   ```

2. **无障碍服务权限**（必需）
   ```xml
   <uses-permission android:name="android.permission.BIND_ACCESSIBILITY_SERVICE" />
   ```

3. **前台服务权限**（用于后台运行）
   ```xml
   <uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
   <uses-permission android:name="android.permission.FOREGROUND_SERVICE_SPECIAL_USE" />
   ```

4. **悬浮窗权限**
   ```xml
   <uses-permission android:name="android.permission.SYSTEM_ALERT_WINDOW" />
   ```

5. **WakeLock 权限**
   ```xml
   <uses-permission android:name="android.permission.WAKE_LOCK" />
   ```

6. **相机权限**（用于二维码扫描）
   ```xml
   <uses-permission android:name="android.permission.CAMERA" />
   ```

#### 权限特点
- ✅ **无需高权限**：不需要 Shizuku 或其他高权限服务
- ✅ **无需root**：所有操作都不需要root权限
- ✅ **标准权限**：只使用 Android 标准权限
- ⚠️ **可能需要查询所有应用权限**：如果需要在 Android 11+ 上查询所有应用，可能需要 `QUERY_ALL_PACKAGES` 权限（但当前实现可能通过其他方式绕过）

## 核心差异总结

### 1. 启动方式差异

**Open-AutoGLM-Android：**
- 单一方案：主要通过无障碍服务启动
- 简单直接，但容错性较低

**当前项目：**
- 多级降级：通用方案 → 品牌特定方案 → ADB命令
- 复杂但容错性高，兼容性好

### 2. 权限差异

**Open-AutoGLM-Android：**
- 可能需要 Shizuku 服务（高权限，需要用户手动安装配置）
- 主要依赖无障碍服务权限
- 使用 Shizuku 可以访问系统级 API，但增加了使用复杂度

**当前项目：**
- 不需要 Shizuku 或其他高权限服务
- 只使用标准 Android 权限
- 所有操作都不需要root权限

### 3. 兼容性差异

**Open-AutoGLM-Android：**
- 通用方案，不针对特定品牌优化
- 在某些品牌设备上可能失败

**当前项目：**
- 针对不同品牌进行优化
- 支持小米、vivo、OPPO、华为、三星等主流品牌
- 在品牌设备上成功率更高

### 4. 容错性差异

**Open-AutoGLM-Android：**
- 如果无障碍服务失败，无法启动应用
- 容错性较低

**当前项目：**
- 如果无障碍服务失败，自动降级到ADB命令
- 容错性高，有多种备选方案

1. ✅ **实现简单**：单一方案，代码简洁

1. ✅ **高兼容性**：多级降级策略，适配不同品牌
