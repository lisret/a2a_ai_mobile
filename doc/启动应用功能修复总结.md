# 启动应用功能修复总结

## 问题分析

从错误日志来看，启动应用失败的主要原因是：

1. **原生模块错误处理不准确**：总是说"input 命令"，但实际上执行的是 `am start` 或 `monkey`
2. **ADB命令权限不足**：`am start` 和 `monkey` 命令在某些设备上需要系统权限
3. **缺少优雅降级**：所有方法都失败后，任务直接失败，没有给用户手动启动的机会

## 修复内容

### 1. 在原生模块中添加启动应用方法

**文件**：`android/app/src/main/java/com/awesomeproject/AccessibilityModule.kt`

**新增方法**：
- `launchApp(packageName: String, promise: Promise)` - 使用Android Intent API启动应用
- `performHome(promise: Promise)` - 执行Home操作
- `getScreenSize(promise: Promise)` - 获取屏幕分辨率

**实现原理**：
```kotlin
// 使用PackageManager获取应用的启动Intent
val pm = reactApplicationContext.packageManager
val intent = pm.getLaunchIntentForPackage(packageName)
intent.addFlags(Intent.FLAG_ACTIVITY_NEW_TASK)
reactApplicationContext.startActivity(intent)
```

**优势**：
- 不需要shell命令，不需要系统权限
- 更可靠，适用于所有Android设备
- 可以检查应用是否已安装

### 2. 修复原生模块错误处理

**文件**：`android/app/src/main/java/com/awesomeproject/AccessibilityModule.kt`

**改进**：
- 根据命令类型（`am start`、`monkey`、`input`）给出不同的错误提示
- 特殊处理退出码137（进程被SIGKILL杀死）
- 错误信息中包含实际执行的命令

**示例**：
```kotlin
when {
    command.contains("am start") -> {
        "启动应用失败：权限不足。am start 命令可能需要系统权限。建议使用无障碍服务启动应用。"
    }
    command.contains("monkey") -> {
        "启动应用失败：权限不足。monkey 命令可能需要系统权限。建议使用无障碍服务启动应用。"
    }
    exitCode == 137 -> {
        "启动应用失败：monkey 进程被系统杀死（退出码137）。可能原因：1. 系统安全策略限制 2. 应用崩溃。"
    }
    // ...
}
```

### 3. 添加优雅降级机制

**文件**：
- `src/services/ADBService.ts`
- `src/services/AccessibilityService.ts`
- `src/services/TaskExecutionHeadless.ts`
- `src/hooks/useTaskExecution.ts`

**实现**：
1. **ADB服务**：当所有启动方法都失败时，抛出特殊错误，标记 `requiresManualLaunch = true`
2. **无障碍服务**：检测到需要手动启动的错误，抛出包含手动启动信息的错误
3. **后台任务执行**：检测到需要手动启动时：
   - 发送 `TaskManualLaunchRequired` 事件到前台
   - 等待用户确认已手动启动（最多30秒）
   - 用户确认后继续执行任务
4. **前台任务执行**：使用Alert提示用户手动启动应用

**新增事件**：
- `TaskManualLaunchRequired` - 需要用户手动启动应用
  - `taskId`: 任务ID
  - `step`: 步骤号
  - `packageName`: 应用包名
  - `message`: 提示消息

- `TaskManualLaunchCompleted` - 用户已手动启动应用
  - `taskId`: 任务ID
  - `step`: 步骤号

### 4. 优化启动方法顺序

**文件**：`src/services/ADBService.ts`

**新的启动顺序**（按优先级）：
1. **Monkey命令**（最优先）- 通常最可靠
2. **Intent + MainActivity** - 使用Intent启动主Activity
3. **直接启动（指定用户）** - 某些设备支持
4. **Intent自动选择Activity** - 让系统选择Activity

## 使用流程

### 正常流程（自动启动成功）

```
无障碍服务启动应用 → 成功 ✅
```

### 回退流程（无障碍服务失败，使用ADB）

```
无障碍服务启动应用 → 失败
  ↓
ADB启动应用（原生Intent API） → 成功 ✅
```

### 优雅降级流程（所有方法都失败）

```
无障碍服务启动应用 → 失败
  ↓
ADB启动应用（原生Intent API） → 失败
  ↓
ADB启动应用（Monkey命令） → 失败
  ↓
ADB启动应用（am start） → 失败
  ↓
发送 TaskManualLaunchRequired 事件
  ↓
等待用户手动启动应用（最多30秒）
  ↓
用户确认已启动 → 继续执行任务 ✅
```

## 优势

1. **更可靠**：使用原生Intent API，不需要shell命令权限
2. **更好的用户体验**：无法自动启动时，提示用户手动启动，而不是直接失败
3. **更准确的错误信息**：根据命令类型给出针对性的错误提示
4. **多级回退**：无障碍服务 → 原生Intent → ADB命令 → 用户手动启动

## 注意事项

1. **原生模块需要重新构建**：添加了新方法，需要重新编译Android应用
2. **权限要求**：原生Intent API不需要特殊权限，但需要应用已安装
3. **用户体验**：如果无法自动启动，用户需要手动启动应用，然后点击"继续"

## 后续优化建议

1. **应用安装检查**：在启动前检查应用是否已安装
2. **启动超时检测**：检测应用是否真的启动成功
3. **应用名称查询**：实现通过应用名称查询包名的功能
4. **启动历史记录**：记录成功启动的应用，下次直接使用
