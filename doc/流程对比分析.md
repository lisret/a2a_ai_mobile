# Open-AutoGLM 流程对比分析报告

## 参考框架流程（Open-AutoGLM）

根据参考框架，核心流程包括：

1. **屏幕感知**：利用视觉语言模型（VLM）解析当前屏幕内容
2. **意图解析**：通过自然语言处理技术理解用户的指令
3. **操作规划**：基于屏幕感知和意图解析的结果，生成具体的操作步骤
4. **执行操作**：通过 ADB 或无障碍服务执行规划的操作
5. **敏感操作确认**：在涉及敏感操作时，系统会请求用户确认
6. **人工接管**：在需要登录或验证码等场景下，系统支持人工接管

## 当前实现流程

当前项目的流程：

1. ✅ **屏幕感知**：通过无障碍服务或ADB截图
2. ✅ **意图解析**：调用模型API解析用户指令
3. ✅ **操作规划**：解析模型响应，提取操作指令
4. ✅ **执行操作**：通过无障碍服务执行操作
5. ❌ **敏感操作确认**：**缺失**
6. ❌ **人工接管**：**部分缺失**

## 发现的问题

### 🔴 严重问题

#### 1. 敏感操作确认机制缺失

**问题描述**：
- 提示词中定义了 `do(action="Tap", element=[x,y], message="重要操作")` 用于处理涉及财产、支付、隐私等敏感操作
- 但代码中**完全没有检测和处理** `message` 参数
- 所有敏感操作都会直接执行，没有用户确认机制

**影响**：
- 安全性风险：可能误操作支付、删除等重要操作
- 不符合参考框架的设计理念

**代码位置**：
- `AutoGLMService.ts:639-660` - Tap操作解析时没有提取message参数
- `TaskExecutionHeadless.ts:520-565` - 执行操作时没有确认机制

**修复建议**：
```typescript
// 在 parseDoAction 中提取 message 参数
case 'Tap': {
  const elementMatch = paramsStr.match(/element\s*=\s*\[\s*(\d+)\s*,\s*(\d+)\s*\]/);
  const messageMatch = paramsStr.match(/message\s*=\s*"([^"]+)"/);
  if (elementMatch) {
    return {
      type: 'click',
      x: mappedX,
      y: mappedY,
      requiresConfirmation: !!messageMatch, // 标记需要确认
      confirmationMessage: messageMatch?.[1] || '此操作涉及敏感信息，是否继续？',
    };
  }
}
```

#### 2. 关键操作类型不支持

**问题描述**：
以下操作在提示词中定义，但代码中**完全不支持**，只是返回 `wait`：

- `Launch` - 启动目标app（提示词规则1要求使用）
- `Interact` - 询问用户如何选择（多选项场景）
- `Take_over` - 人工接管（登录和验证码场景）
- `Note` - 记录页面内容
- `Call_API` - 总结或评论内容

**代码位置**：
```720:728:AwesomeProject/src/services/AutoGLMService.ts
case 'Launch':
case 'Interact':
case 'Note':
case 'Call_API':
case 'Take_over': {
  // 这些操作暂时不支持，返回等待
  console.warn(`[AutoGLM服务] 不支持的操作类型: ${actionType}`);
  return { type: 'wait', duration: 500 };
}
```

**影响**：
- 无法启动目标app，只能通过主屏幕导航（效率低）
- 无法处理需要用户选择的场景
- 无法处理登录/验证码场景（会一直等待）
- 无法记录和总结页面内容

**修复建议**：
- 实现 `Launch` 操作：通过无障碍服务或ADB启动app
- 实现 `Interact` 操作：显示选项列表，等待用户选择
- 实现 `Take_over` 操作：暂停任务，等待用户手动操作后继续
- 实现 `Note` 和 `Call_API` 操作：记录页面内容到任务历史

#### 3. 坐标映射硬编码问题

**问题描述**：
- 代码中硬编码了屏幕分辨率 `1080x1920`
- 实际应该从 native 模块获取真实屏幕分辨率
- 不同设备的分辨率不同，会导致坐标映射错误

**代码位置**：
```647:652:AwesomeProject/src/services/AutoGLMService.ts
// 坐标映射：从 (0-999) 映射到实际分辨率
// 假设标准分辨率为 1080x1920，实际应该从 native 模块获取
const screenWidth = 1080; // TODO: 从 native 模块获取实际屏幕宽度
const screenHeight = 1920; // TODO: 从 native 模块获取实际屏幕高度
const mappedX = Math.round((x / 1000) * screenWidth);
const mappedY = Math.round((y / 1000) * screenHeight);
```

**影响**：
- 在不同分辨率的设备上，点击位置会偏移
- 可能导致操作失败或误操作

**修复建议**：
- 从 AccessibilityService 或 native 模块获取实际屏幕分辨率
- 在截图时记录实际分辨率，并在坐标映射时使用

### 🟡 中等问题

#### 4. Home操作映射不正确

**问题描述**：
- `Home` 操作被映射为 `back`，但两者功能不同
- `Home` 是回到系统桌面，`Back` 是返回上一页

**代码位置**：
```702:705:AwesomeProject/src/services/AutoGLMService.ts
case 'Home': {
  // Home 操作暂时映射为 back（因为我们没有 home 操作）
  return { type: 'back' };
}
```

**修复建议**：
- 实现真正的 Home 操作（通过无障碍服务或ADB）

#### 5. 操作历史上下文维护

**问题描述**：
- 虽然维护了对话历史，但可能没有充分利用操作历史
- 参考框架可能会在提示词中包含操作历史，帮助模型理解当前状态

**修复建议**：
- 在调用模型API时，将最近的操作历史也包含在提示词中
- 帮助模型更好地理解当前状态和下一步操作

### 🟢 轻微问题

#### 6. 错误处理可以更完善

- 连续错误处理机制已实现，但可以增加更详细的错误分类
- 可以区分网络错误、解析错误、执行错误等

## 总结

当前实现基本遵循了参考框架的流程，但在以下关键方面存在缺失：

1. **安全性**：缺少敏感操作确认机制
2. **功能完整性**：多个关键操作类型不支持
3. **准确性**：坐标映射硬编码导致设备兼容性问题

建议优先修复高优先级问题，特别是敏感操作确认机制和关键操作类型的支持，这些是参考框架的核心特性。
