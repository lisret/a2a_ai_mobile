# 优化方案分析报告

## 1. 更新任务记录异步执行

### 当前实现

```typescript
// 当前：阻塞式保存
await taskHistoryService.saveTask(updatedTask);
// 等待保存完成后才继续
await new Promise(resolve => setTimeout(resolve, TASK_TIMEOUT_CONFIG.UI_UPDATE_DELAY_MS));
```

**耗时**：50-200ms（AsyncStorage IO 操作）

### 优化方案

#### 方案1：完全异步（推荐）

```typescript
// 不等待保存完成，立即继续
taskHistoryService.saveTask(updatedTask).catch(error => {
  console.warn('[Headless JS] 保存任务记录失败（不影响任务执行）:', error);
});
// 立即继续，不阻塞
await new Promise(resolve => setTimeout(resolve, TASK_TIMEOUT_CONFIG.UI_UPDATE_DELAY_MS));
```

**优点**：
- ✅ 减少 50-200ms 延迟
- ✅ 不阻塞任务执行流程
- ✅ 任务记录保存失败不影响任务执行

**缺点**：
- ⚠️ 如果应用崩溃，可能丢失最后一步记录
- ⚠️ 需要确保关键节点（完成/失败）的保存是同步的

#### 方案2：关键节点同步，普通步骤异步

```typescript
// 普通步骤：异步保存
if (action.type !== 'complete') {
  taskHistoryService.saveTask(updatedTask).catch(error => {
    console.warn('[Headless JS] 保存任务记录失败:', error);
  });
} else {
  // 完成/失败：同步保存（确保记录）
  await taskHistoryService.saveTask(updatedTask);
}
```

**优点**：
- ✅ 普通步骤不阻塞
- ✅ 关键节点确保保存

**缺点**：
- ⚠️ 代码稍微复杂

### 推荐方案

**采用方案1（完全异步）**，原因：
1. 任务记录主要用于历史查看，不是关键功能
2. 即使丢失一步记录，不影响任务执行
3. 关键节点（完成/失败）已有同步保存

### 性能提升

- **减少延迟**：50-200ms
- **提升比例**：约 3-10%（相对于总耗时 1.5-2.0 秒）

## 2. 截图压缩对耗时的影响

### 当前实现

1. **原生层截图**：获取完整分辨率 Bitmap
2. **转换为 base64**：在原生层完成
3. **传输到 JS**：base64 字符串
4. **发送到 API**：完整图片

**当前截图大小**：
- 1080x1920 分辨率：约 2-3MB（PNG）
- base64 编码后：约 2.7-4MB 字符串

### 压缩方案

#### 方案1：原生层压缩（推荐）

**实现位置**：`AccessibilityModule.kt` 和 `ADBModule.kt`

```kotlin
// 压缩 Bitmap
fun compressBitmap(bitmap: Bitmap, quality: Int = 70): ByteArray {
    val outputStream = ByteArrayOutputStream()
    bitmap.compress(Bitmap.CompressFormat.JPEG, quality, outputStream)
    return outputStream.toByteArray()
}
```

**压缩参数**：
- 格式：JPEG（比 PNG 小 50-70%）
- 质量：70%（平衡质量和大小）
- 分辨率：可选缩放（如 720p）

**耗时分析**：

| 操作 | 不压缩 | 压缩（JPEG 70%） | 压缩+缩放（720p） |
|------|--------|-----------------|-------------------|
| 截图获取 | 1000-2000ms | 1000-2000ms | 1000-2000ms |
| 压缩处理 | 0ms | 50-150ms | 50-150ms |
| base64编码 | 50-100ms | 30-60ms | 20-40ms |
| **总耗时** | **1050-2100ms** | **1080-2210ms** | **1070-2190ms** |
| **图片大小** | 2-3MB | 0.6-1MB | 0.3-0.5MB |

**结论**：
- ✅ **压缩增加耗时：30-110ms**（约 2-5%）
- ✅ **减少传输时间：50-200ms**（网络传输）
- ✅ **减少 API 处理时间**：token 消耗减少 50-70%
- ✅ **总体收益：正收益**（压缩耗时 < 传输节省时间）

#### 方案2：JS 层压缩

**实现位置**：`AutoGLMService.ts`

```typescript
// 使用 canvas API 压缩（需要额外库）
import { manipulateAsync, SaveFormat } from 'expo-image-manipulator';

const compressed = await manipulateAsync(
  screenshotUri,
  [{ resize: { width: 720 } }],
  { compress: 0.7, format: SaveFormat.JPEG }
);
```

**耗时分析**：
- 压缩处理：100-300ms（JS 层较慢）
- 总耗时增加：100-300ms
- **不推荐**：JS 层压缩比原生慢 2-3 倍

### 推荐方案

**采用方案1（原生层压缩）**，具体配置：

1. **压缩格式**：JPEG
2. **压缩质量**：70%（平衡质量和大小）
3. **分辨率**：保持原分辨率（不缩放，避免影响识别精度）
4. **实现位置**：`AccessibilityModule.kt` 和 `ADBModule.kt`

### 性能影响总结

| 指标 | 不压缩 | 压缩（JPEG 70%） | 变化 |
|------|--------|-----------------|------|
| 截图耗时 | 1050-2100ms | 1080-2210ms | +30-110ms (+2-5%) |
| 图片大小 | 2-3MB | 0.6-1MB | -70% |
| 网络传输 | 200-500ms | 60-150ms | -140-350ms |
| API token | 高 | 低 | -50-70% |
| **总体收益** | - | - | **正收益** |

### 注意事项

1. **识别精度**：JPEG 压缩可能略微降低识别精度，但 70% 质量通常足够
2. **分辨率**：不建议缩放，保持原分辨率以确保识别精度
3. **压缩位置**：必须在原生层压缩，JS 层太慢
