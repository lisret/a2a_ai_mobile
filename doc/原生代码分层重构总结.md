# 原生代码分层重构总结

## 完成时间
2024年

## 一、重构目标

将原本混乱的单目录结构（所有文件都在 `com/awesomeproject/` 下）重构为清晰的分层目录结构，提高代码可维护性和可读性。

## 二、新的目录结构

```
com/awesomeproject/
├── bridge/              # React Native 桥接模块
│   ├── AccessibilityModule.kt          # 主桥接模块（包含 ADBModule）
│   ├── AccessibilityActionModule.kt    # 操作桥接模块
│   └── AccessibilityPackage.kt        # 模块注册
│
├── service/             # 服务相关
│   ├── AutoGLMAccessibilityService.kt  # 无障碍服务主类
│   ├── TaskExecutionService.kt         # 任务执行前台服务
│   └── TaskExecutionHeadlessService.kt # Headless JS 服务
│
├── ui/                  # UI 管理模块
│   ├── FloatingWindowManager.kt        # 悬浮窗管理
│   ├── SystemDialogManager.kt          # 系统对话框管理
│   └── NotificationManager.kt          # 通知管理
│
├── accessibility/       # 无障碍功能核心
│   ├── GestureHandler.kt                # 手势处理（点击、长按、滑动等）
│   └── ScreenshotHandler.kt             # 截图处理
│
├── utils/               # 工具类
│   ├── ScreenshotManager.kt             # 截图转换工具
│   ├── WakeLockManager.kt                # WakeLock 管理
│   └── ServiceManager.kt                # 服务管理工具
│
└── core/                # 核心应用类
    ├── MainActivity.kt                  # 主 Activity
    └── MainApplication.kt                # 主 Application
```

## 三、目录说明

### 1. bridge/ - React Native 桥接模块
**职责**: 提供 JavaScript 和 Kotlin 之间的通信接口

- **AccessibilityModule.kt**: 主桥接模块，提供无障碍服务检查、权限管理、悬浮窗、服务管理等接口
- **AccessibilityActionModule.kt**: 操作桥接模块，提供点击、滑动、输入等操作接口
- **AccessibilityPackage.kt**: 模块注册，将桥接模块注册到 React Native

### 2. service/ - 服务相关
**职责**: 管理各种 Android 服务

- **AutoGLMAccessibilityService.kt**: 无障碍服务主类，负责执行自动化操作
- **TaskExecutionService.kt**: 任务执行前台服务，保持应用在后台运行
- **TaskExecutionHeadlessService.kt**: Headless JS 服务，用于后台执行 JavaScript 任务

### 3. ui/ - UI 管理模块
**职责**: 管理各种 UI 组件（悬浮窗、对话框、通知等）

- **FloatingWindowManager.kt**: 悬浮窗的显示、更新和隐藏
- **SystemDialogManager.kt**: 系统级提示窗的显示
- **NotificationManager.kt**: 任务完成通知的显示

### 4. accessibility/ - 无障碍功能核心
**职责**: 实现无障碍服务的核心功能

- **GestureHandler.kt**: 手势操作（点击、长按、双击、滑动）
- **ScreenshotHandler.kt**: 截图功能（支持不同 Android 版本）

### 5. utils/ - 工具类
**职责**: 提供通用的工具方法

- **ScreenshotManager.kt**: Bitmap 到 base64 的转换
- **WakeLockManager.kt**: WakeLock 的获取和释放
- **ServiceManager.kt**: 服务的启动、更新和停止

### 6. core/ - 核心应用类
**职责**: 应用的核心入口类

- **MainActivity.kt**: 主 Activity，处理 MediaProjection 权限请求
- **MainApplication.kt**: 主 Application，注册 React Native 模块

## 九、优势

### 1. 代码组织清晰
- 每个目录职责单一，一目了然
- 文件按功能分类，易于查找
- 目录结构符合 Android 开发规范

### 2. 易于维护
- 修改某个功能时，只需关注对应目录
- 新增功能时，可以明确知道应该放在哪个目录
- 减少代码冲突的可能性

### 3. 可扩展性强
- 每个目录可以独立扩展
- 新增模块时不影响现有结构
- 便于团队协作开发

### 4. 符合最佳实践
- 遵循 Android 项目结构规范
- 模块化设计，职责分离
- 便于单元测试和代码审查

## 十、目录使用指南

### 如何找到文件？

1. **React Native 桥接相关** → `bridge/`
2. **Android 服务相关** → `service/`
3. **UI 组件相关** → `ui/`
4. **无障碍功能实现** → `accessibility/`
5. **工具方法** → `utils/`
6. **应用入口** → `core/`

### 新增文件时应该放在哪里？

- **新的 React Native 模块** → `bridge/`
- **新的 Android 服务** → `service/`
- **新的 UI 管理器** → `ui/`
- **新的无障碍功能** → `accessibility/`
- **新的工具类** → `utils/`
- **应用配置类** → `core/`

## 十一、注意事项

1. **包名必须匹配目录结构**：文件在 `bridge/` 目录下，包名必须是 `com.awesomeproject.bridge`
2. **导入引用必须使用完整包名**：跨目录引用时需要使用完整包名
3. **AndroidManifest.xml 中的类名**：必须使用完整类名（包含包名路径）
4. **服务类名引用**：在代码中引用服务类时，需要使用新的包名路径
