# 模型调用流程检查报告

## 流程概览

### 1. 模型数据存储和获取

**存储位置**: `AsyncStorage` (`@autoglm:models`)

**流程**:
```
用户添加模型 → ModelService.addModel() → saveModels() → AsyncStorage
```

**检查点**:
- ✅ 模型数据正确保存（包含 id, name, apiUrl, apiKey, modelName 等）
- ✅ 模型数据正确读取（getModels()）
- ✅ 错误处理完善

### 2. 模型选择和使用

**流程**:
```
TaskHistoryScreen 加载 → 根据 modelId 获取模型 → 传递给任务执行
```

**代码路径**:
```typescript
// TaskHistoryScreen.tsx
useEffect(() => {
  const loadModel = async () => {
    if (modelId) {
      const loadedModel = await modelService.getModelById(modelId);
      setModel(loadedModel);
    }
  };
  loadModel();
}, [modelId]);
```

**检查点**:
- ✅ 模型根据 modelId 正确加载
- ✅ 如果模型不存在，model 为 null，会有错误提示
- ⚠️ **潜在问题**: 如果 modelId 无效，model 为 null，但错误提示可能不够明显

### 3. 模型传递到任务执行

#### 3.1 前台执行

**流程**:
```
useTaskExecution({ model }) → executeTask() → autoGLMService.getAction(model, ...)
```

**检查点**:
- ✅ 模型正确传递
- ✅ 如果 model 为 null，有错误提示

#### 3.2 后台执行

**流程**:
```
useTaskExecutionWithBackground({ model }) → startBackgroundTask() 
→ JSON.stringify({ model: { id, name, apiUrl, apiKey, modelName, ... } })
→ Headless JS 解析 → executeTaskInBackground(model)
```

**检查点**:
- ✅ 模型数据正确序列化
- ✅ 模型数据正确传递到后台任务
- ⚠️ **潜在问题**: 如果模型数据在序列化/反序列化过程中丢失字段，可能导致问题

### 4. API URL 规范化

**代码位置**: `AutoGLMService.normalizeApiUrl()`

**逻辑**:
```typescript
1. 移除末尾斜杠
2. 如果以 /v1 结尾 → 直接返回
3. 如果以 /v4 结尾或包含 bigmodel.cn → 直接返回（智谱AI）
4. 其他情况 → 添加 /v1
```

**检查点**:
- ✅ 智谱AI URL 正确处理（不添加 /v1）
- ✅ 标准 OpenAI 兼容 API 正确处理（添加 /v1）
- ⚠️ **潜在问题**: 
  - 如果用户输入 `http://127.0.0.1:11434`，会变成 `http://127.0.0.1:11434/v1`
  - 但 Ollama 的默认路径可能不是 `/v1`，需要确认

### 5. API 调用

**代码位置**: `AutoGLMService.getAction()`

**流程**:
```typescript
1. 规范化 API URL: normalizeApiUrl(model.apiUrl)
2. 验证 modelName 不为空
3. 构建请求体（包含截图、对话历史等）
4. fetch(`${apiUrl}/chat/completions`, ...)
```

**检查点**:
- ✅ API URL 规范化
- ✅ 错误处理完善（网络错误、HTTP 错误）
- ✅ localhost/127.0.0.1 特殊提示
- ⚠️ **潜在问题**: 
  - 如果 API URL 配置错误（如 `http://127.0.0.1:11434`），错误提示已改进
  - 但需要确认 Ollama 的正确 API 路径

## 发现的问题

### 问题 1: API URL 规范化可能不正确

**位置**: `AutoGLMService.normalizeApiUrl()`

**问题**: 
- 对于 `http://127.0.0.1:11434`，会规范化成 `http://127.0.0.1:11434/v1`
- 但 Ollama 的 API 路径可能不是 `/v1/chat/completions`，而是 `/api/chat` 或其他路径

**建议**: 
- 如果检测到是 Ollama（端口 11434），使用正确的 API 路径
- 或者让用户明确配置完整的 API URL（包括路径）

### 问题 2: 模型数据验证不够严格

**位置**: `ModelService.addModel()`

**问题**: 
- 只验证了必填字段，但没有验证：
  - API URL 格式是否正确
  - API URL 是否可访问（可选）
  - modelName 格式是否正确

**建议**: 
- 添加更严格的验证
- 在添加模型时，可以尝试连接 API 验证配置是否正确

### 问题 3: 错误消息中的 JSON 可能未正确解析

**位置**: `AutoGLMService.getAction()` 错误处理

**当前实现**: 
```typescript
if (!response.ok) {
  const errorText = await response.text();
  // 尝试解析 JSON
  try {
    const errorJson = JSON.parse(errorText);
    // ...
  }
}
```

**问题**: 
- 如果错误响应是 JSON，但格式不标准，可能解析失败
- 需要更健壮的 JSON 解析

**已修复**: ✅ 已添加更完善的错误解析逻辑

### 问题 4: 模型数据在后台任务中的完整性

**位置**: `useTaskExecutionWithBackground.startBackgroundTask()`

**问题**: 
- 模型数据通过 JSON.stringify 序列化
- 如果模型对象有额外的字段，可能丢失
- 需要确保所有必要字段都包含

**检查**: 
- ✅ 已包含所有必要字段：id, name, apiUrl, apiKey, modelName, description, createdAt, updatedAt

## 建议的改进

### 1. 添加 API URL 验证

在添加/编辑模型时，可以尝试连接 API 验证配置：

```typescript
async function validateApiConnection(apiUrl: string, apiKey: string): Promise<boolean> {
  try {
    const normalizedUrl = normalizeApiUrl(apiUrl);
    const response = await fetch(`${normalizedUrl}/models`, {
      method: 'GET',
      headers: {
        'Authorization': `Bearer ${apiKey}`,
      },
    });
    return response.ok;
  } catch {
    return false;
  }
}
```

### 2. 改进错误消息

对于 localhost/127.0.0.1 错误，提供更详细的解决方案：

```typescript
if (apiUrl.includes('127.0.0.1') || apiUrl.includes('localhost')) {
  throw new Error(
    `无法连接到本地服务 ${apiUrl}。\n\n` +
    `在 Android 设备上，127.0.0.1 指向设备本身，无法访问开发机器上的服务。\n\n` +
    `解决方案：\n` +
    `1. 如果使用本地服务（如 Ollama），请使用开发机器的实际 IP 地址\n` +
    `   例如：http://192.168.1.100:11434/v1\n` +
    `2. 获取开发机器 IP 的方法：\n` +
    `   - Windows: ipconfig（查找 IPv4 地址）\n` +
    `   - macOS/Linux: ifconfig 或 ip addr show\n` +
    `3. 确保设备与开发机器在同一 Wi-Fi 网络\n` +
    `4. 如果使用云端服务，请检查 API URL 配置是否正确`
  );
}
```

### 3. 添加模型配置测试功能

在模型列表页面，添加"测试连接"功能，允许用户测试模型配置是否正确。
