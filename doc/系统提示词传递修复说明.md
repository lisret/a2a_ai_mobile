# 系统提示词传递修复说明

## 问题发现

在检查模型调用环节时，发现系统提示词只在第一步传递，后续步骤不会传递系统提示词。

### 问题代码

```typescript
// 添加系统提示词
if (isFirstStep && request.systemPrompt) {
  messages.push({
    role: 'system',
    content: request.systemPrompt,
  });
}
```

### 问题影响

1. **模型可能忘记系统规则**：在后续步骤中，模型可能忘记系统提示词中定义的规则和约束
2. **动态内容无法更新**：系统提示词可能包含动态内容（如搜索框位置），如果只在第一步传递，后续步骤无法获取最新信息
3. **不符合LLM API最佳实践**：大多数LLM API建议在每次调用中都包含系统提示词

## 修复方案

### 修改内容

将系统提示词从"只在第一步添加"改为"每一步都添加"：

```typescript
// 添加系统提示词（每一步都需要，确保模型始终记住规则）
// 注意：虽然对话历史中可能包含系统消息，但为了确保系统提示词始终在消息列表的第一位，
// 并且是最新的（可能包含动态内容如搜索框位置），我们在每一步都添加系统提示词
if (request.systemPrompt) {
  messages.push({
    role: 'system',
    content: request.systemPrompt,
  });
}
```

### 修复理由

1. **确保模型始终记住规则**：系统提示词定义了模型的行为规则、操作指令格式、约束条件等，模型需要在每一步都记住这些规则

2. **支持动态内容更新**：系统提示词可能包含动态内容，如：
   - 搜索框位置（可能在不同步骤中更新）
   - 当前日期（虽然通常不变，但理论上可能变化）
   - 其他运行时信息

3. **符合LLM API标准**：大多数LLM API（如OpenAI、Anthropic等）都建议在每次API调用中都包含系统提示词

4. **避免上下文丢失**：虽然对话历史中可能包含之前的系统消息，但：
   - 系统消息在对话历史中被过滤掉了（`filter(msg => msg.role !== 'system')`）
   - 即使不过滤，也无法保证系统提示词在消息列表的第一位
   - 无法保证系统提示词是最新的（包含动态内容）

### 性能影响

- **Token消耗**：每次API调用都会包含系统提示词，会增加token消耗
- **影响评估**：
  - 系统提示词通常较长（几千token）
  - 但这是必要的，因为模型需要在每一步都记住规则
  - 相比模型忘记规则导致的任务失败，这个token消耗是值得的

### 验证方法

1. **日志验证**：在开发模式下，检查每次API调用的消息列表，确认系统提示词是否包含
2. **功能验证**：测试多步骤任务，确认模型在后续步骤中仍然遵循系统提示词中的规则
3. **动态内容验证**：测试搜索框位置更新后，模型是否能正确使用新的搜索框位置信息
